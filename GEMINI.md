## 1. 운영 제1원칙: 사전 승인 시스템 (Strict Approval)
- **독단적 수정 금지**: 어떠한 파일도 에이전트 임의로 수정하거나 생성하지 않는다.
- **승인 절차**: 모든 파일 변경 작업은 반드시 사전에 **[기획 및 설계안 보고] -> [사용자 명시적 승인]** 절차를 거쳐야 한다. 사용자로부터 "진행해", "수정해" 등 명확한 승인이 있을 때만 도구를 사용한다.
- **자의적 판단 지양**: 설계가 모호하거나 판단이 필요한 경우, 스스로 결정하지 않고 즉시 사용자에게 질문하여 의도를 파악한다.

## 2. 실무 개발 워크플로우 (Core Workflow & Engineering Rationale)

본 프로젝트의 통제 구조를 최우선으로 하며, 시뮬레이터의 물리적 정확성과 코드 품질을 위해 다음의 세부 지침을 준수한다.

### 2.1 작업 단위의 철저한 분리 (Task Granularity)
- **컴포넌트 단위**: '4분할 워크스페이스(Layout)', '모달(UI)', '편집기(Editor)' 등 독립된 시각적 단위로 쪼개어 구현한다. 이는 UI 렌더링 부하를 최소화하고 재사용성을 높이기 위함이다.
- **기능 단위**: '수식 파싱($V$)', 'Solver 엔진', '시각화(Plotting)'를 분리할 뿐만 아니라, **1D와 2D 구현 단위를 명확히 분리**하여 개발한다. 특히 물리 법칙이 코드로 변환되는 과정에서 발생할 수 있는 오류를 차단하기 위해, 연산 로직은 UI 프레임워크와 완전히 격리한다.

### 2.2 체크리스트 기반의 루프 실행 (Loop-based Execution)
- **선 설계 후 구현**: 작업을 시작하기 전, 수행할 태스크의 체크리스트를 먼저 작성한다.
- **최소 파일 원칙**: 현재 루프에서 해결해야 할 기능에 꼭 필요한 파일만 생성하거나 수정하여, 코드 베이스가 불필요하게 비대해지는 것을 방지한다.
- **단일 책임 원칙 (SRP)**: 하나의 함수는 오직 하나의 물리적/논리적 기능만 수행한다. 이는 복잡한 슈뢰딩거 방정식 풀이 과정에서 디버깅 효율을 극대화하기 위함이다.

### 2.3 예외 처리 및 검증 (Exception & Validation)
- **테스트 케이스 기반 검증**: 기능 구현 후 테스트 코드를 작성하거나 Well-known 케이스(예: 무한 우물, 조화 진동자 등)를 통해 물리적 타당성을 검증한 후 다음 단계로 이동한다.
- **실시간 오류 브레이크**: 로컬 환경(`npm run dev` 등)에서 개발 중 에러가 발생하면, 즉시 프로세스를 중단(break)하고 사용자에게 알린다. 이는 잘못된 로직이 누적되어 전체 시뮬레이션의 신뢰도를 떨어뜨리는 것을 막기 위한 핵심 방어 기제다.
